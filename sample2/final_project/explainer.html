Motivation

The datasets we have choosen for our project are the NYPD Motor-Vehicle-Collision and the NY NOAA Weather dataset:

  • data.cityofnewyork.us/Public-Safety/NYPD-Motor-Vehicle-Collisions
  • www.ncdc.noaa.gov/cdo-web/datasets
The NYPD dataset contains over 1 million rows of data with multiple rows of same date value and 29 features, with the NOAA dataset containing the same amount of rows as dates in the NYPD dataset and 4 features.

Motivation for this project started with the idea of cross referencing the NY vehicle accident dataset with the NY weather dataset, and see how much of the data correlates to the number of accidents and type of accidents. With this data we hoped that certain trends would emerge visually and give an overview of the most dangerous areas in NY with regard to the traffic. This means we could potentially help warn trafficants of accident prone areas and be extra vigilant.

Our end goal was to have the user explore the data with regard to seasonal weather and the accidents in the different boroughs. We put more focus on a visualized dashboard rather than a narrative story since we hope that seeing how and why accidents happen in the different boroughs would send a message of driving accordingly to the weather conditions.

Basic stats

Starting with the NYPD dataset containing traffic collisions we ran a check of data completeness in python with the pandas library, to see how much data was missing from each feature. Here we see the first 7 features in which we concentrated on making a complete dataset.

  • DATE is 100.00% complete
  • TIME is 100.00% complete
  • BOROUGH is 71.36% complete
  • ZIP CODE is 71.35% complete
  • LATITUDE is 82.57% complete
  • LONGITUDE is 82.57% complete
  • LOCATION is 82.57% complete

As we see, latitude and longitude has around 17.5% missing data. The dataset is filtered by removing all rows where the data is not complete, In this case we remove all observations with latitude or longitude missing.

  • DATE is 100.00% complete
  • TIME is 100.00% complete
  • BOROUGH is 83.08% complete
  • ZIP CODE is 83.08% complete
  • LATITUDE is 100.00% complete
  • LONGITUDE is 100.00% complete
  • LOCATION is 100.00% complete

This is repeated with zip code and borough, such that we only work with observations that do not miss borough, zip code, or latitude/longitude. Similar processing is applied to the NOAA weather data, such that we only keep observations for the same dates that we also have collison data on. As we found out later, the data actually need further scrutiny, since as "Main accident factors" (See Data story) in the NYPD dataset contained empty values ' ' and 'Unspecified'. These values are in of itself also data but it turned out to classify almost a 1/4 of the NYPD data and does then not properly explain the reason of the accident. Whether or not to keep this data is hard to evaluate but as of now we have decided to keep it. This shows atleast that for all the incidents alot of them have not been properly recorded in the dataset and can be a cause of wrongly skewed data.

After all the preprocessing we ended up with a total of 425.992 observations/data points.

Genre

For our visual narrative we have:

  • Consistent visual platform
  • Feature distinction
  • Animated transitions
We wanted a strong visual dashboard with rich opportunities, therefore we decided for a consistent visual platform for the user to explore. We chose many distinct features such as different plots and filtering across many groups to increases the explorability of our visualizations, and together with animated transitions this should spark curiosity in the user.

And for our narrative structure we have:

  • Random access
  • Filtering / Selection / Hover Highlighting / Details
  • Caption / Headlines

Our narrative allows for highly random access, as the narrative is told from the highly configurable and unstructured visualization dashboard. The interactivity of our narrative comes from the filtering, selecting, details and highlighting of our visualizations. The site is mostly defined by captions and headlines, and these make up the overall structure of the website.

Visualizations

As we explained in the motivation, we put a heavy focus on a comprehensive and interactive visualization dashboard. We wanted to have a dashboard where the user was free to play around and explore for them selves. We decided for the following visualizations on the dashboard:

  • Doughnut chart
  • Bar chart
  • Choropleth
  • Timeline (Bar chart)
The doughnut chart is used to display the top 10 contributing factors of the traffic accidents. A doughnut chart is useful for visualizing the percentage distribution of few to medium amounts of categories, especially if the categories are not ordinal. The barchart plots the number of collisons per borough and a barchart is more fitting for this as we have few categories with quantitative data. The choropleth is a visual pleaser, and gives a sense of the distribution of collisions geographically. Finally the timeline is represented by a barchart, which is the optimal choice. Furthermore, we wanted ALL the figures to interact with each other, such that the user has maximum ability to filter and explore the data. We chose these four different charts, because they cover most of the visualization types we learned about in this course. Also, having many different types of charts allows the visualizations to extract more varied information out of the data.

These complex and dynamic visualizations and the dashboard is fitting for our data story, as we seek a detailed look into the collisions of new york, where the user has a lot of control of what they want to explore. Less focus has been put on telling a straight chronologic story, where the user instead would have their hand held all the time.

Discussion

Even though we purposely put heavy focus on advanced and comprehensive visualizations, and less on a chronologic data story, we would admittedly have liked to have a more indepth story. Our "data story" is more like a data analysis/exploration that where we interpret our visualizations and correlation figures.

Secondly, we aimed to have high performance visualizations with as many observations as possible. To achieve this we explored the Javascript library Crossfilter, which enables fast filtering of large datasets in the browser. This library can be used with D3.js to construct high performance visualizations. Combining Crossfilter and D3 was harder than originally thought, but then we found the library dc.js. DC.js is a library build on D3.js with native Crossfilter support. This library made it easier for us to combine Crossfilter and D3.js

Despite the our several visualizations, there are still more visualizations we would have liked to include. We especially wanted a stacked bar chart of pedestrains, cyclists and motorists, and how many was injured/killed in the selected collisions, but we did not manage to get it to work. Initially we also planned to have a scatter plot of all collisions on top of a geographial chart, however, with 425.992 observations this was simply not possible, so we went with a choropleth instead. Furthermore, we are missing visualizations of the weather data, as we would have liked the possibility to filter collisions by weather condition. These would no doubt have made our narrative better and more interesting, but we simply ran out of time to implement this functionality, and instead the weather data is only used for correlation analysis.
